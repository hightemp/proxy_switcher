# Traffic statistics screen

## Problem

The app had no way to see how much traffic passed through the local proxy or whether connections were being opened, closed, or failing.

## Design

Traffic accounting belongs in `ProxyServer`, not in the UI or foreground service. The proxy core is the only layer that knows byte direction:

- `client -> upstream` is upload
- `upstream -> client` is download

The new `ProxyStats` singleton exposes a `StateFlow<ProxyStatsSnapshot>` with session counters. It resets on proxy start and remains visible after stop, so the user can inspect the last session.

## Solution

- Added byte counters for uploaded/downloaded traffic.
- Added session counters for active tunnels, total tunnels, peak active tunnels, failures, uptime, and total traffic.
- Added a `Statistics` screen showing total traffic, upload/download, average rates, active/peak tunnels, total tunnels, and failures.
- Added a Home toolbar entry for the Statistics screen.
- Added focused unit tests for `ProxyStats`.
- Updated `AGENTS.md` so future agents know the project now has 7 routes and a stats module.

## Prevention

- Keep runtime counters near the subsystem that owns the measured events.
- Count proxy payload bytes, not UI events or persisted preferences.
- Use ADB UI dumps for device-level smoke checks of new screens.
