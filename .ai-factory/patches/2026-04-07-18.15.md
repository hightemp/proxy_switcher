# IPv4 loopback proxy bind fix

**Date:** 2026-04-07 18:15
**Files:** `ProxyServer.kt`, `TASKS.md`
**Severity:** high

## Problem

After starting the proxy on a physical Android device, apps had no internet even though the foreground service was running and system proxy was set to `127.0.0.1:8080`.

## Root Cause

`ProxyServer.createLoopbackServerSocket` used `InetAddress.getLoopbackAddress()`. On the tested Samsung device this resolved to IPv6 loopback `::1`, so the server listened on `[::1]:8080`. Android system proxy settings used IPv4 `127.0.0.1:8080`, so Chrome and other apps connected to an address where nothing was listening.

ADB confirmed the mismatch:

- Before fix: `ss -ltnp` showed `[::1]:8080`
- `toybox nc -4 -z -w 3 127.0.0.1 8080` returned `Connection refused`
- `toybox nc -6 -z -w 3 ::1 8080` connected

## Solution

Changed the local proxy bind address to explicit IPv4 loopback: `InetAddress.getByName("127.0.0.1")`.

## Verification

- Installed debug build on `SM-A165F - 15` through ADB
- Started proxy through the app UI
- Verified `settings get global http_proxy` returned `127.0.0.1:8080`
- Verified `toybox nc -4 -z -w 3 127.0.0.1 8080` connected
- Verified `ss -ltnp` showed `[::ffff:127.0.0.1]:8080`
- Verified HTTPS proxy CONNECT returned `HTTP/1.1 200 Connection Established`
- Opened `https://example.com` in Chrome through active system proxy and saw no `ERR_PROXY_CONNECTION_FAILED`
- Stopped proxy through UI and verified system proxy restored to `:0`
- Ran `GRADLE_USER_HOME=/tmp/gradle ./gradlew testDebugUnitTest lintDebug`

## Prevention

- Keep proxy listener bind address and system proxy host identical.
- Do not rely on platform-dependent loopback address selection for user-visible network settings.
- For Android device verification, test both socket listener address and browser/system proxy behavior.
