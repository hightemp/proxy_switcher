# Home button runtime status fix

**Date:** 2026-04-07 18:06
**Files:** `HomeScreen.kt`, `ProxyViewModel.kt`, `ProxyService.kt`, `ProxyRuntimeState.kt`, `ProxyRuntimeStateTest.kt`, `TASKS.md`
**Severity:** medium

## Problem

On a physical Android device, the Home screen could show the wrong proxy action: `START PROXY` while the proxy was already running, or `STOP PROXY` while it was stopped.

## Root Cause

The Home screen state was initialized from the persisted `KEY_WAS_RUNNING` preference. That preference is a START_STICKY restart flag, not a reliable live service status. If it became stale or if the screen missed a status broadcast, the UI could render the opposite action.

## Solution

Added `ProxyRuntimeState`, a process-local `StateFlow` for the foreground service's actual runtime status. `ProxyService.sendStatus` updates this state whenever it broadcasts service status, and `ProxyViewModel` collects it instead of reading the persisted restart flag as live UI state.

## Verification

- `GRADLE_USER_HOME=/tmp/gradle ./gradlew testDebugUnitTest lintDebug`
- `GRADLE_USER_HOME=/tmp/gradle ./gradlew installDebug`
- `adb shell pm grant com.hightemp.proxy_switcher android.permission.WRITE_SECURE_SETTINGS`
- `adb shell pm grant com.hightemp.proxy_switcher android.permission.POST_NOTIFICATIONS`
- `adb shell am start -n com.hightemp.proxy_switcher/.MainActivity`
- `adb shell input tap 540 1540`
- `adb shell uiautomator dump /sdcard/proxy_switcher_ui.xml`

Device verification showed `STOPPED` + `START PROXY` before start, `RUNNING` + `STOP PROXY` after start, and `STOPPED` + `START PROXY` after stop.

## Prevention

- Do not use restart/recovery preferences as live UI state.
- Prefer a single runtime status source for same-process service/UI synchronization.
- Keep device-level UI checks for foreground service state changes.
